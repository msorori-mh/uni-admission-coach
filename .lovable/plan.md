

## ملخص الخطة

إعادة تصميم مسار تسجيل الدخول بحيث يسجل الطالب مرة واحدة فقط (عبر Google أو رقم الجوال)، ثم يُكمل بياناته الأكاديمية، وبعدها يدخل مباشرة دون تسجيل دخول متكرر.

## التدفق المطلوب

```text
أول مرة:
  صفحة Login (Google / Phone) → تسجيل دخول ناجح
    → فحص: هل أكمل الطالب بياناته الأكاديمية؟
      → لا → صفحة CompleteProfile (محافظة، جامعة، كلية، تخصص)
      → نعم → Dashboard

المرات التالية:
  الجلسة محفوظة تلقائياً → Dashboard مباشرة
```

## التغييرات المطلوبة

### 1. إنشاء صفحة `CompleteProfile` جديدة
- **ملف جديد**: `src/pages/CompleteProfile.tsx`
- تعرض خطوتين:
  - **الخطوة 1**: الاسم الأول، اللقب، المحافظة
  - **الخطوة 2**: الجامعة → الكلية → التخصص (قوائم ديناميكية من قاعدة البيانات)
- عند الإرسال: تحديث جدول `students` بالبيانات المختارة
- بعد الحفظ: توجيه إلى `/dashboard`
- تتحقق أولاً من وجود جلسة نشطة، وإن لم تكن موجودة توجه لـ `/login`

### 2. تعديل صفحة `Login.tsx`
- إبقاء زر Google وزر رقم الجوال كما هي
- **إزالة** أي روابط لصفحة التسجيل (`/register`)
- بعد نجاح تسجيل الدخول بـ Google: فحص بيانات الطالب في جدول `students`
  - إذا `major_id` فارغ → توجيه لـ `/complete-profile`
  - إذا مكتمل → توجيه لـ `/dashboard`

### 3. تعديل `useAuth.ts`
- بعد تحميل بيانات المستخدم، فحص اكتمال الملف الشخصي
- إذا كان الطالب لم يكمل بياناته الأكاديمية (`major_id = null`) والصفحة الحالية ليست `/complete-profile` → توجيه تلقائي

### 4. تعديل `App.tsx`
- إضافة مسار `/complete-profile` يشير لصفحة `CompleteProfile`
- إزالة مسار `/register` (لم يعد مطلوباً لأن التسجيل يتم عبر Google/Phone مباشرة)

### 5. تعديل صفحة `Index.tsx`
- تغيير أزرار "إنشاء حساب" و"تسجيل الدخول" لتشير كلها لـ `/login`

### 6. حذف `Register.tsx`
- لم تعد مطلوبة — التسجيل يتم تلقائياً عبر trigger `handle_new_user` عند أول دخول بـ Google

## ملاحظات تقنية
- الـ trigger الموجود `handle_new_user` يُنشئ سجل الطالب تلقائياً عند أول تسجيل دخول بـ Google (مع `major_id = null`)
- الجلسة محفوظة تلقائياً في `localStorage` عبر إعداد `persistSession: true` الموجود في Supabase client
- لا حاجة لتغييرات في قاعدة البيانات
